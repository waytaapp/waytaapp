Mobile UI/UX Audit · WCAG 2.2 AA
Wayta Order & Pay PWA · v2.2 · All roles · Exhaustive

A fast engine wearing an unreadable interface.

The architecture is genuinely strong — offline sync queues, role-based navigation, safe-area handling, first-use QR guidance. But the surface works against it: a micro-typography system (7–10px uppercase labels) that fails legibility on every screen, sub-44px touch targets throughout, a payment gesture that is physically broken on most phone widths, no way to add an item to the cart from the menu, and "terminal-core" copy that reads as alarming jargon to a patron standing in a loud, dark venue. The findings below are grouped per screen with annotated recreations, then rolled up into an action plan.

Section 1

Critical mobile findings

You cannot add an item to the cart

Menu item cards have no add button and are not tappable — there is no path from browsing to ordering. The cart total shown in the footer is a hardcoded value. This is the revenue path, and it is severed.

Slide-to-pay is broken off a single screen width

The drag distance is a fixed 280px constant. On narrower phones the handle can't reach the end; on wider ones payment triggers early. There is also no tap alternative — a drag-only payment gesture excludes switch, screen-reader, and motor-impaired users.

Staff order board uses desktop drag-and-drop

The kanban relies on HTML5 drag-and-drop, which does not fire on touch screens. Bartenders on phones or tablets cannot move an order to "Ready" — the core throughput workflow fails on the devices it's built for.

7–10px uppercase micro-type is the house style

Nav labels, menu descriptions, stat cards, toasts, and status text render at 7–10px with heavy letterspacing and uppercase. That is below any legibility floor — in a dark venue, one-handed, slightly drunk, it is unreadable. Minimum body/label size should be 12–13px, 11px only for true captions.

Checkout has no exit and no confirmation

The checkout screen has no back button — the only escape is the bottom nav. After paying, the app fires an email and redirects after a timeout, with no in-UI receipt or success moment. With an empty cart, the pay slider is still live at R0.

Fabricated data erodes trust in a payments app

The order ETA is a fake countdown that always shows ~10 minutes. Review counts, vendor stats, "25% better night satisfaction", and simulated biometric success ("hardware resonance scan") all present invented numbers as real. In an app that takes money, any discovered fake datum poisons all real ones.

Section 4

Screen transition & motion strategy

Adopt one spatial model

Views currently cross-fade with slight vertical drift — every navigation feels the same, so users build no mental map. Use: lateral slide for drill-ins (venue → menu → checkout) with edge-swipe back; bottom-sheet rise for transient tasks (filters, item detail, budget); crossfade only for bottom-nav tab switches. Back should always reverse the entry motion.

Shared-element continuity

Venue card → venue page: let the card image expand into the header (View Transitions API works in a PWA). Menu item → detail sheet: the thumbnail grows into the sheet's hero. The cart bar's total should tick numerically when items are added, with a small badge pop on the cart icon.

Skeletons, not spinners

Replace the blocking "Connecting to bar..." spinner and full-screen loaders with layout-true skeletons: venue-card ghosts on Explore, row ghosts on the menu, a status-card ghost on tracking. Images should progressive-load with a dominant-color placeholder — never pop from empty gray.

Haptics & success moments

Vibration API: light tick on add-to-cart and stepper taps; medium buzz when slide-to-pay completes; success pattern when the order flips to "Ready" (paired with a push notification). Payment success deserves a real moment — a drawn checkmark and receipt card, not an 800ms redirect. Kill the ambient animations (pulsing shields, glowing QR) so motion regains meaning.

Section 5

Suggested mobile additions

Missing states
  • Empty cart (currently a void with a live pay slider)
  • Order-failed / payment-declined state with retry
  • Post-payment receipt & confirmation screen
  • Menu skeleton + image placeholders
  • "Venue closed / not accepting orders" state
  • Staff board "no orders yet tonight" state
Missing components
  • Item detail bottom sheet (size, modifiers, notes)
  • Persistent scan FAB / center nav slot
  • Filter & sort bottom sheet (replacing selects + dropdown)
  • Undo snackbar for destructive actions
  • Sticky menu category tabs with scroll-spy
  • Manual code entry + torch toggle in QR scanner
Structural moves
  • Split Profile: move the inbox/email client out; keep wallet, payment methods, settings
  • Wire browser back/deep links to the view stack so Android back works
  • Hide global bottom nav inside the order flow (single contextual bar)
  • One semantic color per order state, app-wide
  • Copy pass: replace Pulse/Mesh/Grid/Node vocabulary with plain words
Section 6

What's already working

  • Offline detection with a sync queue and status banner — rare and valuable for venue Wi-Fi dead zones
  • Safe-area padding on the bottom nav; backdrop-blur sticky bars
  • QR scanner two-step first-use guide with skip
  • Auth progress bar and OTP resend countdown
  • Role-based navigation shaping (guest vs patron vs staff)
  • Empty states exist on Tickets and Explore (they need copy + contrast work, but the pattern is there)
  • Order-status stepper concept is right — it just needs honest data
  • Image error fallbacks on venue/menu cards
